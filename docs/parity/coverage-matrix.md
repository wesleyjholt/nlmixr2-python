## Overall assessment

The Python port currently covers a narrow package-surface bootstrap, not the substantive modeling-and-estimation capabilities of upstream `nlmixr2`.

Practical summary:

- Package-entry helper names: mostly present
- Model DSL: lightly sketched
- Native estimation: absent
- Fit-object semantics: mostly absent
- Post-fit utilities and controls: absent
- Upstream ecosystem integration: absent or only represented as metadata

## Matrix

| Area | Upstream `nlmixr2` capability | Python status | Notes |
| --- | --- | --- | --- |
| Package entry points | `nlmixr2`, `nlmixr2CheckInstall`, `nlmixr2conflicts`, `nlmixr2deps`, `nlmixr2packages`, `nlmixr2update` exported from the R package namespace | Partial | Names are present, but Python semantics are simplified environment helpers, not full R behavior |
| Model entry/inspection | `nlmixr2(object)` can parse/check models before fitting and returns a uniform model object | Partial | Python can build and return a structured model object, but not a parsed RxODE/nlmixr model with upstream classification/inspection behavior |
| `ini` block semantics | Numeric initials, bounds, reserved-name restrictions, variance-scale conventions | Partial | Python supports scalars, `c(est)`, `c(lower, est)`, `c(lower, est, upper)`-like tuples and reserved-prefix validation |
| Random-effects specification | `eta` / omega syntax using `~`, including correlated blocks | Missing | No Python support for variance/covariance declarations or correlated random effects |
| Residual-error specification | `add()`, `prop()`, combined residual error, `norm()` | Missing | Python `model()` is plain strings with no residual-error semantics |
| ODE model DSL | RxODE-style differential equations inside the model block | Missing | No parser, solver integration, or event handling in the Python package |
| Solved PK systems | `linCmt()` solved systems for 1/2/3 compartment models with absorption/lag support | Missing | No solved-system support |
| Covariates / mu-referencing | Upstream model conventions for covariates and SAEM-friendly parameterization | Missing | No covariate semantics beyond raw input columns |
| Dataset semantics | Upstream nlmixr/rxode2-compatible event datasets | Partial | Python accepts columnar or record data but only checks shape consistency |
| Censoring support | `DV`, `CENS`, `LIMIT` conventions and censored residual handling | Missing | No censoring-aware likelihood or output |
| Estimator dispatch | Upstream `nlmixr2()` dispatches to estimation backends and methods listed by `nlmixr2AllEst()` | Missing | Python only supports `est="mock"` and `est="reference"` |
| FOCEi estimation | Native FOCEi fitting and its controls | Missing | Python can ingest one FOCEi artifact generated by upstream R, but does not perform FOCEi |
| SAEM estimation | Native SAEM fitting and its controls | Missing | Explicitly absent |
| NLME / NLM / other estimators | Upstream estimator/control support visible in docs and helper pages | Missing | No native Python implementation |
| Estimation controls | `foceiControl()`, `saemControl()`, `nlmControl()`, `nlmixr2NlmeControl()`, `tableControl()` | Missing | Python accepts `control`/`table` dicts but does not implement the control constructors or semantics |
| Fit object richness | Upstream fit carries objective info, covariance/correlation, theta/eta outputs, timing, per-subject information, data-frame-like behavior | Missing | Python fit object contains only estimator, observation count, columns, parameter count, objective, model, control, and table payloads |
| Objective-function handling | Upstream `setOfv()` and related AIC/BIC/objective type behavior | Missing | No equivalent |
| Post-fit utilities | `bootstrapFit()`, `traceplot()`, fit-print/inspection helpers | Missing | No equivalent |
| Ecosystem package checks | Upstream install checks report core and optional R packages | Partial | Python reports Python package versions and one reference-runtime version, but not upstream optional-package semantics |

## What Is Actually Covered Today

Covered with tests in this repo:

- top-level helper names listed above
- a minimal `ini()` representation
- a minimal `model()` representation
- `nlmixr2()` returning either:
  - a Python model object
  - a mock fit summary from supplied data
  - a reference-backed fit loaded from a Gautschi-produced FOCEi artifact

This means the current package is best described as:

- a parity harness
- a model-spec container
- a reference-artifact consumer

It is not yet a usable replacement for upstream `nlmixr2` model fitting.

## Highest-Priority Missing Work

If the goal is substantive parity rather than namespace parity, the biggest missing items are:

1. Native estimator support, starting with FOCEi or SAEM
2. A real model DSL that captures random effects, residual error, and ODE / solved-PK semantics
3. Proper control objects and fit-object structure
4. More Gautschi parity artifacts across multiple models and estimators
5. Data semantics beyond rectangular column ingestion, especially event-table and censoring support

## Recommended sequencing

Recommended next implementation order:

1. Expand reference-artifact coverage first so native work has parity targets
2. Define a stricter Python model IR for fixed effects, random effects, residual error, and solved-PK/ODE branches
3. Implement one native estimator path with parity tests
4. Fill in control objects and richer fit outputs around that estimator
5. Add post-fit utilities after at least one estimator is real

## Confidence notes

- The statement that the Python port is not close to full `nlmixr2` parity is high confidence and directly supported by the current repo state.
- The categorization of some upstream capabilities under the umbrella package is based on official `nlmixr2` docs and reexport/helper pages. Where the docs point through to `nlmixr2est`, the practical user-facing capability still exists upstream but is not currently implemented in this Python repo.
